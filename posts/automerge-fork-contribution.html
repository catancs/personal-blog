A first pull request to Automerge

11 June 2026 View the PR →

The last thing I wrote here was about a rich-text CRDT that started life as a chapter in Martin Kleppmann's database book and ended with me accidentally reinventing an award-winning algorithm. I thought that was the end of the thread. It wasn't. Pulling on one curiosity has a way of fraying into three more, and one of those loose ends turned into the small thing I want to write about now: my first pull request to Automerge.

It happened the way these things usually do. While I was buried in Eg-walker papers and CRDT benchmarks, Kleppmann's name kept surfacing, and his projects with it. One of them was Automerge, a library I'd been quietly leaning on as a mental reference point the whole time without ever having opened the hood. It's a serious piece of work: a JSON-like data structure that several people can edit at once and merge back together automatically, with a compact binary format and a sync protocol underneath. The tagline they use is "PostgreSQL for your local-first app," and that's roughly the ambition. A core written in Rust, compiled to WebAssembly, wrapped for JavaScript and C. Maintained out of Ink & Switch.

Wanting to give something back

Spending weeks inside this corner of computing and taking so much from it, the papers, the reference designs, the libraries I measured my own toy against, left me wanting to put something back in the other direction, however small. So I did the unglamorous thing first. I read the contributing notes, I read the open issues, and I went looking specifically for the future-work list, the stuff the maintainers have flagged but haven't gotten to.

Most of what I found was deep core work, the kind of thing where a careless change quietly costs you a 19x performance regression and three days of someone else's time. Not where a newcomer should be making his first move. But tucked in among the issues was a much friendlier one: a list of command-line tools the project knows it's missing, sitting in an IDEAS.md file that had been there for years. Forking a document at a particular point in its history. Diffing two documents. Getting and setting values by path from the terminal. None of it built yet.

That was the right size. Self-contained, genuinely wanted, and far enough from the delicate parts that I could be confident I wasn't breaking anything subtle. I picked the one that mapped most cleanly onto machinery that already existed: fork.

The change itself

The honest description is that this is a small contribution, and I'd rather say that plainly than dress it up. fork takes a document and a change hash and produces a new document as it existed at that point in history. The reason it was a good first pick is that the core library already knows how to do the hard part, fork_at is right there in the Rust API. The CLI just didn't expose it. So the work was mostly plumbing done carefully: read from a file or standard input, write to a file or standard output, reuse the verification flag the other commands already use, and fail with a clear message when someone hands it a hash that doesn't exist instead of producing a broken file.

While I was in there I found a small piece of dead code: an old command file that referred to an internal API the project had removed long ago. It couldn't compile, nothing referenced it, and it was the only reason two dependencies were still being pulled in. So the pull request deletes it and drops those dependencies, in its own separate commit so it's easy to see and easy to reason about. The net result is a change that adds a feature and removes more lines than it adds, which is my favourite shape for a patch.

The part I actually cared about

If the last project taught me anything it's that the discipline matters more than the code. I'm building all of this by orchestrating agents, and the entire risk of that way of working is accepting a green checkmark you haven't earned. So I held this to the same standard, even though it's a fraction of the size.

I built it from a clean baseline so I could prove nothing was broken before I touched it. I wrote integration tests that drive the actual compiled binary: fork at an earlier point and confirm the later edits are gone, fork at the latest point and confirm everything's still there, fork at a hash that isn't in the document and confirm it refuses to write anything. I ran the formatter and the linter the project's own CI runs, and got them clean. And I split the work into three commits, each touching one thing, with messages that explain why rather than what, because that's what the contributing guide asks for and because the next person to read git blame deserves it. None of that is heroic. It's just the difference between offering someone tidy work and offering them homework.

Where it stands

The pull request is open, and before opening it I left a note on the relevant issue saying which piece I was taking and that I was deliberately scoping it to just fork rather than bundling in the diff and get/set commands, which are bigger and deserve their own attention. Now I wait. The maintainers are two people doing this largely full-time alongside a community, and a first-time contributor's PR sits behind a manual approval before the test suite even runs, which is a perfectly reasonable thing to gate on. They might merge it, they might ask me to change the shape of it, they might point out something I missed. Any of those is a fine outcome. I'll update this if and when it lands.

Mostly I wanted to mark the moment, because there's a neat symmetry to it. A database textbook sent me down a rabbit hole that ended with me reinventing one of its author's algorithms by accident. The same rabbit hole has now ended with me opening a small pull request against another of his projects, on purpose this time. Taking ideas from people you admire is easy. Trying to hand a little something back, even something this modest, feels like the better end of the deal.
